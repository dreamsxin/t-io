  • 新版本说明

    • 添加了大量更简洁的API
      • 异步发送切换为阻塞发送只需要把Aio.sendXxx()改成Aio.bSendXxx()即可，接近零成本
      • 提供了分页获取连接列表的功能（特别适合im应用中的用户列表场景）
      具体见下图红框部分
    • 用tio实现了部分httpserver协议，并且附带了一个类似springmvc用法的路由功能----根据请求path找相应方法
    • 用tio实现了websocket协议(pong协议部分，个人觉得是协议设计得不合理，所以改由客户端来维护心跳)，并且附带了一个类似springmvc用法的路由功能----根据自定义消息体的某个字段找相应方法
    • voovan作者发布了t-io的ab test结果，引发了t-io对短连接场景的初步支持（以前也是支持的，不过作者一直强调tio是长连接框架，短连接场景性能不会太好）----提醒：voovan作者测试结果发布时 t-io尚未官方宣传支持短连接，所以数据仅供参考对比
    • t-io生态圈项目成立：目前已经放置tio-httpserver和http-websocket，对tio性能各种质疑的可以用一个公有协议对tio进行各种测试了----“短连接”场景短时间内不会优化太多，用作参考就好， 下图是作者亲测数据，有兴趣和有质疑的可以测试验证，测试步骤在 tio httpserver ab test

    • t-io官方主群不交保护费就会被踢出群----群里有一群优秀的开源作者和技术主管级成员，还有OSC官方人员监场，这个谣言竟然还传开了，弄得群里几个人找我交保护费
    • 给t-io差评会被删除----OSC目前并未提供这样的功能
    • t-io是netty的一个壳而已----t-io并没有引用netty作为底层实现，3000多行代码以及pom都摆在那哦
    • t-io收费传闻----t-io项目本身不会收费，协议也只会是LGPL，但是用t-io的项目可以收费
    • t-io的高性能完全是jdk的功劳，t-io只是作了封装而已----想告诉这位朋友，所有用jdk写的东西都有jdk的功劳
    • t-io只适合小白级用户----小白用户容易上手并且能做到生产环境中还极其稳定，反而说明框架健壮性（保证程序运行稳定）和抽象性（保证API容易使用）好，高手用起来会更顺手，目前有不少用t-io实现的各类项目，譬如IM项目、游戏项目、物联网项目、http服务器（已经实现主要功能）、websocket服务器（已经完整实现）等，相信这根本不是小白能玩得转的
